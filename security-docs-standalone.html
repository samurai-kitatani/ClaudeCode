パスワードリセット機能に穴がある\n\n#### 対策\n- パスワードは `bcrypt` または `Argon2` でハッシュ化する\n- セッションにはランダムな十分長いIDを使う\n- ログアウト時にサーバー側でセッションを破棄する\n\n```javascript\n// ❌ ダメな例\nconst hash = md5(password); // MD5は解読可能\n\n// ✅ 正しい例\nconst bcrypt = require('bcrypt');\nconst hash = await bcrypt.hash(password, 12); // ソルト付きハッシュ\n```\n\n---\n\n### 5｜認可の不備（権限昇格）\n\n#### どんな攻撃か\nログインはできる状態で、他人のデータや管理機能にアクセスできてしまう問題。\n\n```\n【水平権限昇格の例】\nユーザーA（ID:100）がログインした状態で\nhttps://example.com/api/user/101/data\nにアクセス → ユーザーB（ID:101）のデータが見えてしまう\n\n【垂直権限昇格の例】\n一般ユーザーが\nhttps://example.com/admin/delete-user\nにアクセスできてしまう\n```\n\n#### AIエージェント開発で起こりやすいケース\nAIが「動く機能」を生成することに集中し、「誰がアクセスできるか」のチェックが抜けることが多い。\n\n#### 対策\n- APIの各エンドポイントで「このユーザーはこのデータにアクセスできるか」を必ず確認\n- ミドルウェアで認証・認可をまとめて処理する\n\n```javascript\n// ✅ 各APIで所有者チェック\napp.get('/api/order/:id', authMiddleware, async (req, res) => {\n const order = await Order.findById(req.params.id);\n \n // 自分の注文だけアクセスできる\n if (order.userId !== req.user.id) {\n return res.status(403).json({ error: 'Forbidden' });\n }\n res.json(order);\n});\n```\n\n---\n\n### 6｜APIキー・シークレットの漏洩\n\n#### どんな攻撃か\nGitHubなどにコードを公開した際に、APIキーやパスワードが含まれていて \n攻撃者に悪用される。または元社員・元メンバーがキーを持ち続ける問題。\n\n#### よくある事例\n- `.env` を `.gitignore` に書き忘れてGitHubに公開\n- OpenAI APIキーが公開されて数十万円の請求が来る\n- 退職したエンジニアのAPIキーが無効化されていない\n\n#### 対策\n- `.env` を必ず `.gitignore` に追加する\n- GitLeaks でスキャンを習慣化する\n- APIキーは**環境（本番・開発）ごと**に分ける\n- 担当者が変わったらキーをローテーション（更新）する\n\n---\n\n### 7｜サプライチェーン攻撃\n\n#### どんな攻撃か\n信頼しているライブラリ（npm・pip等）が悪意あるコードを含んでいる、 \nまたはAIが存在しないパッケージ名を勧めてくる（ハルシネーション）問題。\n\n#### AIエージェント特有のリスク\n**AIが存在しないパッケージ名を推薦することがある（約20%）**。 \nそのパッケージ名と同じ名前で攻撃者が悪意あるパッケージを公開していることがある。\n\n```bash\n# ❌ 危険な流れ\nAIが推薦 → 存在しないパッケージ名 → 攻撃者が同名で公開 → インストールしてしまう\n\n# ✅ 対策\n- パッケージをインストールする前にnpmjs.com/PyPI で実在確認\n- ダウンロード数・更新日・公式ドキュメントの記載を確認\n- npm audit を定期的に実行\n```\n\n---\n\n### 8｜機密情報・個人情報の漏洩\n\n#### どんな攻撃か\n意図せずデータベースの内容・ユーザー情報・ログが外部に露出する。\n\n#### よくあるパターン\n- エラーメッセージにDBのテーブル名・スタックトレースが含まれる\n- デバッグログが本番環境で有効のまま\n- バックアップファイルがWebからアクセスできる状態\n\n#### 対策\n- 本番環境では**詳細なエラーをユーザーに返さない**（ログに記録するだけにする）\n- 開発用のデバッグ設定を本番で有効にしない\n- `.bak`、`.sql`、`.log` ファイルをWebルートに置かない\n\n---\n\n### 9｜SSRF（サーバーサイドリクエストフォージェリ）\n\n#### どんな攻撃か\nサーバーを経由して、攻撃者が指定した内部URLや外部URLにリクエストを送らせる攻撃。 \nクラウド環境では内部のメタデータサービスにアクセスされることがある。\n\n```\n攻撃者 → 「このURL（http://169.254.169.254/...）にリクエストして」\n → サーバーがAWSの内部メタデータAPIにアクセス\n → IAM認証情報が漏洩\n```\n\n#### AIエージェント開発で起こりやすいケース\n「ユーザーが指定したURLの内容を取得する」機能でURLを検証せずに使う場合。\n\n#### 対策\n- ユーザーが指定できるURLを**ホワイトリスト**（許可リスト）で制限する\n- 内部ネットワークアドレス（192.168.x.x、10.x.x.x 等）へのリクエストをブロック\n\n---\n\n### 10｜DoS・レート制限なし\n\n#### どんな攻撃か\n大量のリクエストを送ることでサーバーをダウンさせる攻撃。 \nAIエージェントの場合、大量にAI APIを呼ばれることで**巨額の課金**が発生する可能性もある。\n\n#### 対策\n- APIにレート制限を設ける（例：1ユーザーあたり60回/分まで）\n- AI API呼び出しに上限を設ける（月次コスト上限設定）\n\n---\n\n### 11｜ハルシネーション依存攻撃\n\n#### AIエージェント固有の脅威\n\nAIが「存在しないライブラリ・API・コード」を自信満々に提案することで \nそれを信じて組み込んでしまい、セキュリティの穴が生まれる。\n\n#### 具体例\n- 存在しないセキュリティ関数を使う → 実際にはセキュリティ処理がない\n- 廃止された脆弱なライブラリを推奨される\n- 間違った暗号化方式を「安全です」と提案される\n\n#### 対策\n- セキュリティに関わるコードは**公式ドキュメントで必ず確認**する\n- 「このコードは本当に存在しますか？」と二重確認する習慣\n- 暗号化・認証などの重要部分は信頼できるライブラリの公式ドキュメントに従う\n\n---\n\n### 12｜Memory Poisoning（AIエージェント固有）\n\n#### どんな攻撃か\nAIエージェントが長期記憶（メモリ）を持つ場合、 \nその記憶に悪意ある情報を注入し、将来の行動を操作する攻撃。 \nWebページや文書から情報を取得するエージェントで特に注意が必要。\n\n#### 対策\n- エージェントが取得・記憶できる情報源を制限する\n- 記憶の更新には人間の確認を入れる\n- エージェントが参照する外部ドキュメントを信頼できるソースに限定する\n\n---\n\n## クライアントへの説明：攻撃の難易度イメージ\n\n```\n攻撃者のレベル感（低い方が多い攻撃者でも実行可能）\n\n初心者でも可能 ←────────────────────→ 高度なスキルが必要\n │ │\nAPIキー漏洩 XSS プロンプト SSRF Memory\nSQLインジェクション 認可不備 インジェクション サプライチェーン Poisoning\n\n※ 初心者でも可能な攻撃ほど、対策の優先度が高い\n```\n\n---\n\n## 対策の優先順位（まず何から手をつけるか）\n\n### フェーズ1：絶対にやる（リリース前必須）\n1. **APIキーを `.env` に移す** → 1時間以内にできる\n2. **パスワードをハッシュ化する** → ライブラリ1つで対応\n3. **SQLはプリペアドステートメント/ORM を使う** → コード修正\n4. **認可チェックを全APIに入れる** → 設計の見直し\n\n### フェーズ2：早めにやる\n5. **プロンプトインジェクション対策**（入力とシステムプロンプトを分離）\n6. **レート制限を入れる**\n7. 「AIを使って素早く開発できるのは大きな強みです。一方で、スピードが上がった分、セキュリティのチェックを意識的に組み込まないと、穴が生まれやすくなります。今日はその穴を事前につぶすための話をしたいと思います。」\n\n---\n\n#### 認証・認可\n\n- **認証**（Authentication）：「あなたは誰か」を確認する仕組み（ログイン）\n-
